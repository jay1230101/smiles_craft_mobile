Smile Craft Mobile

Milestone 3 — Delivery Summary

Project: Smile Craft mobile app
Platforms: iOS & Android
Milestone: M3 — Patient Records, Appointment Actions & Profile
Status: Delivered

1. What this milestone covers

Milestone 3 turns the app from a read-only schedule viewer into a working clinic tool. Clinic staff can now register new patients, cancel and reschedule appointments from a tap-and-act popover on the calendar, manage their account from a proper profile screen, and review a live notifications feed of everything happening in the clinic.

2. Features delivered

2.1 Patient Registration Live
  • New-patient form on the Register tab with first name, last name, father's name, date of birth, phone, gender, assigned doctor, email, and allergies/notes.
  • Built-in validation: required-field checks, date-of-birth is auto-masked to DD-MM-YYYY and rejected if not a real past date, phone is digits-only, email is checked for the right format.
  • Duplicate detection: if the phone is already registered to another patient, the app surfaces a confirm dialog showing the existing patient's name and gives the user the choice to continue anyway or cancel.
  • Role-aware doctor selection: non-owner doctors can only register patients under their own name (the field is locked); owners, assistants, and admins pick from a doctor list.
  • Wired to the existing POST /register-patient backend; gender list pulled live from GET /getGender.
2.2 Calendar Appointment Popover Live

Tapping a booked appointment on the Day or Week view opens a popover modal with patient name, status, time range, treatment, and doctor.

  • Cancel appointment — reason picker (loaded live from GET /cancellation-reasons), submits to POST /cancel-appt, and the calendar updates immediately when the server confirms.
  • Edit / Reschedule — opens a full-screen edit form pre-filled with the appointment's current date, start & end time, doctor, treatment, and a WhatsApp-reminder toggle. Saves through POST /encounter (the backend's update path).
  • Confirm appointment — shown as a placeholder action. Needs a small backend addition (see Section 4) before it can be wired.
  • Charge / Take payment — shown as a placeholder. Ships with the Billing module in M4 (the backend endpoint, /treatment-plan, handles both the encounter recording and the billing in one call).
  • Already-cancelled appointments show the Cancel action greyed out with explanatory helper text.
2.3 Edit / Reschedule Appointment Live
  • Modal-style screen reached from the appointment popover's "Edit / Reschedule" action.
  • Editable: date (DD-MM-YYYY mask), start & end time (HH:MM mask), doctor (picker), treatment, and a WhatsApp-reminder toggle.
  • End time must be after start time; date and time fields are validated as the user types.
  • Patient identity (name, family, DOB, phone) is carried over from the selected appointment so the backend's patient lookup matches.
  • On save, the calendar and dashboard refresh automatically to show the new schedule.
2.4 Profile & Settings Live
  • New profile screen reached from the avatar button in the Dashboard header (replaces the interim Logout button that was sitting on the dashboard since M2).
  • User card with avatar (auto-initials), name, role (pretty-printed: "Owner Doctor", "Assistant", etc.), and email.
  • Preferences section with Language and Push Notifications listed as roadmap items (push needs a backend update — see Section 4).
  • About section with the app version, build number, and clinic ID.
  • Log out button with a confirmation dialog before signing out.
2.5 Notifications Inbox Live
  • New Notifications screen reached from the bell button on the Dashboard.
  • Captures every appointment and patient change that arrives in real time over the existing socket connection (new bookings, updates, confirmations, cancellations, deletions, patient registrations, edits, removals).
  • Each entry has a kind-aware icon and tint, the headline of what changed, who it relates to, and a relative timestamp ("Just now", "5m ago", "3h ago", etc.).
  • The Dashboard bell shows a red dot as long as there are unread notifications.
  • Mark all as read and Clear all actions; the inbox keeps the 100 most recent items.
  • This is a client-side feed; a server-backed inbox would land later if the team wants notifications to persist across devices or app reinstalls.

3. Technical notes

  • All M3 screens are fully wired to the existing live backend — no new backend services were stood up. The mobile app uses the same APIs the web app already uses.
  • Demo mode (which the QA tester uses by default) returns realistic fake responses so every flow can be exercised without hitting the live system. Switching from demo to live is a one-line flag.
  • Forms use React Hook Form + Zod (same stack as the M2 login and forgot-password forms) for consistent validation and a single way to maintain forms going forward.
  • The doctor picker added in the M2 polish round is reused throughout M3 — same selection control on Calendar, Dashboard, Registration, and Edit screens.

4. Backend updates needed for the next milestone

Two small backend additions would close the placeholders shown in M3:

  • Manual confirm appointment endpoint — today the only way an appointment moves to "confirmed" is through the WhatsApp webhook (patient taps Confirm in WhatsApp). The mobile app needs:
    POST /confirm-appt  { bookingId }  →  sets patient_confirmed = True and emits confirmedAppointment on the same rooms the existing flow uses.
  • Push notification device-token registration — needed before iOS/Android push notifications can be turned on:
    POST /register-device-token  { token, platform }  →  stores per-user, then the existing socket emits should call a Celery task that pushes to those tokens via FCM / APNs.
